    From tool to agent

    Four slides on the shift from chatbot to agent: what changed, what it means, and where agents are being deployed.

    Slide 1 of 4

    The chatbot era is ending

    For most of AI's recent history, interaction meant a single exchange: you send a message, the model replies. The model does not act. It does not plan. It waits for the next input. This model is useful but limited. An increasing number of AI deployments in 2025 and 2026 involve agents: systems that take a goal and execute a series of steps to accomplish it, with minimal human intervention at each step.

    Slide 2 of 4

    What an agent actually is

    An agent is a language model equipped with tools and the ability to plan multi-step tasks. It can call a search API to get current information. It can run code, read files, send emails, query databases, or interact with websites. It decides which tool to use, when to use it, and what to do with the result. It iterates until it reaches the goal or runs out of steps. The model is the reasoning engine. The tools are its hands.

    Slide 3 of 4

    Why this changes everything

    When a chatbot gives a bad answer, you read it and discard it. When an agent takes a wrong action, it may have already sent the email, deleted the file, or placed the order. The shift from output to action changes the stakes fundamentally. It also changes the evaluation criteria: correctness of reasoning is necessary but not sufficient. Reliability, reversibility, and scope of action all become critical design decisions.

    Slide 4 of 4

    Where agents are being deployed now

    Agents are being deployed in software development (writing, testing, and deploying code), customer service (resolving tickets end to end), research (gathering and synthesizing information from multiple sources), and operations (scheduling, reporting, and workflow automation). The common pattern: tasks that previously required a human to execute a sequence of well-defined steps, where the steps can be made legible to a model.

    Prompting and tool use

    Four mechanisms that make agents work. Click each to expand.

    A prompt is everything that goes into the model's context window before it generates a response: the system instruction, the conversation history, any retrieved documents, any tool results. Prompt engineering is the practice of structuring this input to elicit the desired output. It sounds simple and can become surprisingly complex. System prompts define agent behavior, persona, and constraints. Few-shot examples (examples of desired behavior included in the prompt) often outperform lengthy instructions.

    Tool use means the model can invoke external functions and receive their results. The model generates a structured tool call (for example: search query, code to execute, or API call), the framework executes the call, and the result is appended to the context window. The model then continues generating based on the result. Tool use extends the model from a text predictor into a system that can gather information, take actions, and interact with external services. It is the mechanism that turns a chatbot into an agent.

    Chain-of-thought prompting asks the model to show its reasoning step by step before giving a final answer. This improves performance on complex tasks because the model can catch and correct errors in intermediate steps. Planning goes further: the model breaks a high-level goal into subtasks, decides the order of execution, and handles dependencies between steps. Models like OpenAI's o1 and o3 are specifically trained for extended chain-of-thought reasoning, producing significantly better results on difficult multi-step problems.

    A multi-agent system uses multiple language models working in concert. An orchestrator model receives the high-level goal and delegates subtasks to specialized worker models. A research agent gathers information. A writing agent drafts content. A review agent checks for errors. Each agent has a specific role, tool access, and context. Multi-agent architectures allow parallelism and specialization that a single model cannot achieve. They also introduce new failure modes: cascading errors, conflicting outputs, and coordination overhead.

    1. Observe

    The agent receives input: the user's goal, conversation history, results from previous tool calls, retrieved documents, or the state of the system it is operating on. Everything the agent knows is in its context window. Observation is not passive: the agent may call tools to gather information as part of this step.

    2. Plan

    The model reasons about the goal and decides the next action. With chain-of-thought prompting, this reasoning is explicit and visible. The plan may involve a single next step or a sequence of steps. More capable models produce more reliable plans. Planning is where model quality matters most: a weak model makes poor plans that cascade into wrong actions.

    3. Act

    The agent executes the planned action by calling a tool or generating output. The action might be a web search, a code execution, an API call, or a message. The framework intercepts the tool call, executes it in the environment, and returns the result. This is the step where agents differ most fundamentally from chatbots: they take actions with real-world consequences.

    4. Evaluate

    The agent reviews the result of the action and decides whether the goal has been achieved, whether to continue, or whether to change approach. This is a critical step. A poorly designed agent may continue past the point where a task is complete, or stop too early. Human-in-the-loop checkpoints can be inserted here for high-stakes decisions.

    Agent vs chatbot

    What makes an agent fundamentally different from a chatbot?

    Actions, not answers

    A chatbot produces text output. An agent takes actions: it calls tools, executes code, makes API requests, and interacts with external systems. A chatbot's output is a draft. An agent's output is often a side effect in the real world. This difference changes everything: error handling, verification, reversibility, and oversight all become critical design concerns.

    Tool use

    How does a language model use a tool?

    Structured call, returned result

    The model generates a structured tool call describing what it wants to do (search for X, execute this code, call this API). The framework intercepts the call, runs it in the environment, and returns the result to the model as additional context. The model continues generating based on the result. Tool use is mediated by the framework, not executed by the model directly.

    Chain of thought

    What is chain-of-thought reasoning and why does it help?

    Visible intermediate steps

    Chain-of-thought prompting asks the model to show its reasoning before giving a final answer. This helps because errors in intermediate steps can be caught before they affect the conclusion. It is also how o1 and o3 models work: they perform extended internal reasoning before outputting a response. Visible reasoning is also easier to audit, which matters for high-stakes applications.

    MCP

    What is the Model Context Protocol?

    A standard for tool connections

    MCP (Model Context Protocol) is an open standard developed by Anthropic for connecting language models to external tools and data sources. It defines how tools describe their capabilities and how models request tool calls. MCP is analogous to what USB was for hardware: a standard connector that allows any model to use any compatible tool. Adoption is growing across the AI ecosystem.

    The transition from chatbot to agent is the most consequential shift currently underway in how AI is deployed. It is not a gradual improvement in text quality. It is a change in what the system does: from producing output to taking action. That change has implications for reliability, oversight, and risk that do not apply to chat interfaces.

    Tool use is the mechanism. Chain-of-thought reasoning is the method. Multi-agent systems are the architecture. Together, these three components describe how most serious AI deployments work today, and how almost all of them will work in the near future.

    The question for anyone working near AI is not whether agents will be deployed in their domain. It is how to design the oversight, reversibility, and error-handling that makes agent deployment safe. That is the design problem of the next several years.
