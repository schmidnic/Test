Context, memory, and limits

1 / 8

    The context window

    Every language model operates within a context window: the maximum amount of text it can process in a single interaction. Everything inside the window is visible to the model. Everything outside is invisible. The model cannot recall it, reference it, or reason about it. The context window is not RAM. It is not a cache. It is the complete scope of the model's attention for that interaction.

    Slide 2 of 4

    Why context windows were small for so long

    Attention in a transformer scales as O(n²): compute grows with the square of the number of tokens. Double the context, quadruple the compute. This made long contexts expensive and slow. Early GPT models had 4,096 tokens, roughly three to four pages. Today, leading models support 1 million tokens or more. The architectural innovation that enabled this is an active area of research, including the sparse attention approaches discussed in the transformer microlearning.

    Slide 3 of 4

    Memory is not what you think it is

    When a model seems to remember something from earlier in a conversation, it is because that earlier text is still in the context window. When the window fills up, older content is dropped. When you start a new session, the context resets entirely. There is no persistent memory by default. The model does not know you from last time. Every session is a new entity with no history.

    Slide 4 of 4

    What this limits in practice

    A model cannot read an entire novel in one pass. It cannot reason over a year of project notes. It cannot maintain continuity across a series of sessions without external help. Every AI assistant that appears to have memory is using something beyond the base model: a database, a summarization system, or a retrieval pipeline. Understanding this gap is essential for building and evaluating AI systems honestly.

    Retrieval Augmented Generation. When a user sends a query, relevant content is retrieved from an external knowledge base and inserted into the context window alongside the query. The model then generates a response using both the query and the retrieved content. RAG allows models to reason about documents they were not trained on, to access current information, and to reduce hallucination by grounding answers in retrieved evidence. It is the dominant technique for giving models access to external knowledge.

    An embedding is a dense numerical vector representing the meaning of a piece of text. Similar texts have similar vectors. A vector database stores millions of these embeddings. When a query arrives, it is converted to an embedding, and the database finds the stored embeddings most similar to it. These are the most semantically relevant chunks of text. They are retrieved and inserted into the model's context window. This is how RAG finds the right content to retrieve.

    A vector database is a storage system optimized for similarity search on high-dimensional vectors. Unlike a traditional database that looks up rows by exact value, a vector database finds the nearest neighbors of a query vector in a high-dimensional space. Popular vector databases include Pinecone, Weaviate, and Qdrant. They are a core component of RAG systems. The quality of retrieval, which depends on embedding quality and indexing, determines how useful the retrieved content is.

    Several approaches extend beyond the context window. Summarization compresses long histories into shorter representations. Retrieval stores conversation history externally and retrieves relevant parts when needed. External tools give models access to structured storage like databases or files. Long-context models push the limit of what fits. Each approach has tradeoffs in accuracy, cost, and latency. No single approach fully solves the memory problem, and most production systems combine several.

    Embedding the query

    The user's query is converted into an embedding: a dense vector representing its meaning. This vector is used to search the vector database for semantically similar content. The quality of the embedding model determines how well the retrieval step finds relevant material.

    Vector search

    The vector database contains embeddings for all the documents in the knowledge base. The nearest neighbors of the query embedding are retrieved: the chunks of text most semantically similar to the query. Typically the top 3 to 10 chunks are retrieved and inserted into the context window.

    Language model

    The retrieved chunks and the original query are both placed in the context window. The model generates a response using both. Because the relevant information is now in the context window, the model can answer based on the retrieved evidence rather than relying on what it learned during training. This significantly reduces hallucination for factual questions.

    Grounded response

    The model's response is grounded in retrieved evidence. It can cite specific passages. It has access to information outside its training data. And because the evidence is provided explicitly in context, the model is less likely to confabulate. RAG does not eliminate hallucination entirely, but it substantially reduces it for factual retrieval tasks.

    Context

    What happens when a conversation exceeds the context window?

    Earlier content is lost

    When the context window fills, the model cannot process all the text at once. Typically, older content is dropped from the beginning of the conversation. The model has no access to what was dropped. This is why long conversations can lose coherence, and why AI assistants that need to track extended history use external storage rather than relying on context alone.

    RAG

    What problem does RAG solve that a larger context window does not?

    Cost, freshness, and scale

    A large context window still has limits and costs. RAG retrieves only the relevant portion of a large knowledge base, inserting just what is needed. It can access documents not in the training data. It can be updated in real time without retraining. And it scales to knowledge bases far larger than any context window. The two approaches are complementary, not alternatives.

    Embeddings

    Why are embeddings useful for retrieval?

    Semantic similarity

    Embeddings represent meaning as vectors. Semantically similar texts have similar vectors, even if they share no keywords. A query about "fixing a leaking faucet" can retrieve a document about "plumbing repair" because their embeddings are close in vector space. This goes beyond keyword matching and enables retrieval based on meaning.

    Memory

    How do AI assistants that remember past conversations actually work?

    External storage, not model memory

    The base model has no persistent memory. Assistants that remember you are using an external system: a database that stores conversation history, a summarization pipeline that compresses it, and a retrieval system that injects relevant past content into the current context window. The model reads its own history the same way it reads any other retrieved document.

    What to take with you

    The context window is not just a technical limit. It defines the scope of the model's attention. Everything inside it is visible. Everything outside is gone. Understanding this prevents the most common category of errors in AI system design: expecting models to track information they cannot see.

    RAG is the dominant solution for knowledge retrieval, but it is not a complete solution. Retrieval quality depends on embedding quality. Relevant chunks can be missed. The model can still hallucinate when retrieved context is ambiguous or insufficient. RAG reduces the problem significantly without eliminating it.

    Context length is growing fast. What was architecturally impossible two years ago is now routine. The combination of longer contexts and better retrieval is making AI systems capable of tasks that seemed permanently out of reach. The limit is moving, and knowing where it currently sits is part of staying current.
