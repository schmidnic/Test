Agency

May 10, 2026 · Written By Nicole Schmidbauer

Did I build this?

I have known what I wanted this website to look like for about three years. The ghost word behind the headline. The typography that felt like a Sunday morning, not a productivity app. The way the hero section should breathe without shouting.

What I didn't have was the technical ability to build it. I tried. Page builders, templates, "no-code" tools that turned out to have quite a lot of code hiding behind them. Every time I got close to the vision, something blocked me: a CSS property I didn't understand, a layout that refused to behave, a font that looked right in theory and wrong in practice. I ended up with compromises. Not bad, but not it.

Then I tried Claude Code. In one weekend session, I had the website I had imagined. Now I have to be honest about what that means.

How the setup works

The setup is deliberately simple and almost free. Three components.

A GitHub repository holds all the website files: HTML, CSS, and JavaScript, no framework, no build step. Claude Code works directly with these files in your local terminal.

Claude Code is Anthropic's command-line tool. You run it in your project folder, describe what you want in plain language, and it reads, writes, and edits your files directly. It can make a targeted change ("make this font slightly smaller") or a comprehensive one ("redesign this section to feel lighter and more editorial"). It explains what it's doing and asks before anything irreversible.

Netlify connects to your GitHub repository and deploys your site automatically every time you push a change. Free tier, custom domain, HTTPS: done in about ten minutes.

The cost: Claude Code charges per API use, roughly a few euros for a full design session. Netlify and GitHub are free at this scale. What you need is a GitHub account, a terminal you're not afraid of, and a clear idea of what you want. That last requirement turns out to be the hard part.

What the process actually looked like

The hardest part was not technical. It was articulation.

Claude Code can do a great deal, but it can only do what you can describe. My first prompts were vague: "make it feel more editorial," "I want it to look like a real publication." These produced results, but not quite the right results. Iterating felt like the design conversations I imagine happen in real agencies, except I was simultaneously the client and the person trying to translate client wishes into something buildable.

At some point I realised: I was learning to have taste in language. To say "the ghost word should overflow the right edge, with the heading aligned to the bottom-left, and the overall height reduced so the overflow appears at the top as well." That sentence took me twenty minutes to formulate. Before this project, I would not have known what I was looking at, let alone how to ask for it.

The technical decisions were made by Claude Code. The aesthetic decisions were made by me. The translation between them was the actual work.

I had expected the process to feel like dictating to a very fast assistant. It felt more like learning a new language for describing visual ideas, with a patient conversation partner who could immediately show me what my words produced.

Back to the question I keep asking

In "The effortless illusion" I wrote about desirable difficulty: the friction that builds competence, the struggle you can't afford to skip.

I skipped learning CSS deeply. I don't feel conflicted about that. CSS is a means to an end, and for what I needed — a personal blog with a specific aesthetic — the end was the point, not the means.

What I didn't skip was the work of knowing what I wanted. That is harder than it sounds. Most people, given a blank canvas and infinite tools, produce something mediocre: not because they lack taste, but because translating internal vision into external decisions is a skill that develops slowly, through years of looking at things and noticing what works and why. I have been working on that skill for a long time. Claude Code gave me a tool to finally use it.

I also want to be honest about what I still don't understand. I can navigate the code now and make small changes. But I couldn't have written it from scratch, and I don't understand every decision embedded in the structure. That gap is real. I have chosen to accept it in the same way I accept not understanding the combustion engine that moves my car.

What this means for skill development in the AI age

The more interesting question is not "did you build this" but "what did building this require?" The answer is: clarity of vision, the ability to describe aesthetic choices precisely, the patience to iterate, and enough taste to know when something is right versus almost right. Those are not the skills that show up in job descriptions for front-end developers.

Many technical capabilities are becoming accessible through language. What is becoming more valuable, not less, is the ability to know what you want and say it well — to develop enough judgment that you can direct a powerful tool toward a specific outcome rather than just generating output. The answer to "what should I develop" is no longer "whatever the tutorial teaches next." It's closer to: develop deep taste, learn to articulate it, stay curious about which tools can extend your reach.

I'm writing this on the website I imagined three years ago. The typography feels like a Sunday morning. The hero breathes.

I built this — with help I couldn't have built it without. And asking honestly what that means is the right question to sit with as AI becomes the context in which we all figure out what we are actually capable of.
